# Аутентификация в API

REST API проекта Be cooking использует ту же систему аутентификации, что и основной HTML-интерфейс приложения.

Аутентификация реализована через Flask-Login и session cookie. После входа пользователя в систему Flask сохраняет информацию о пользователе в серверной сессии, а браузер получает cookie `session`.

## Общий принцип

Пользователь сначала проходит авторизацию через обычную страницу входа сайта.

После успешного входа браузер хранит cookie сессии. При последующих запросах к API эта cookie автоматически отправляется вместе с запросом.

API использует `current_user`, чтобы определить, авторизован ли пользователь и какие действия ему доступны.

## Тип авторизации

В API используется session-based authentication.

В OpenAPI это описывается как cookie-based authentication:

```yaml
securitySchemes:
  cookieAuth:
    type: apiKey
    in: cookie
    name: session
```

Это означает, что защищённые endpoint-ы ожидают cookie с именем session.

## Защищённые endpoint-ы

Некоторые API endpoint-ы доступны всем пользователям, а некоторые требуют авторизации.

**Публичные endpoint-ы:**
- `GET /api/recipes` | Получить список рецептов
- `GET /api/recipes/{recipe_id}` | Получить один рецепт

Эти endpoint-ы доступны без входа.

**Endpoint-ы, требующие авторизации:**
- `POST /api/recipes` | Создать рецепт
- `PATCH /api/recipes/{recipe_id}` | Обновить рецепт
- `DELETE /api/recipes/{recipe_id}` | Удалить рецепт

Для этих операций пользователь должен быть авторизован.

## Проверка авторизации

Для API используется отдельный декоратор `api_login_required`.

Он похож на стандартный `@login_required`, но вместо редиректа на HTML-страницу входа возвращает JSON-ошибку.

**Пример логики:**

```python
def api_login_required(view_func):
    @wraps(view_func)
    def wrapper(*args, **kwargs):
        if not current_user.is_authenticated:
            return api_error("Необходима авторизация.", 401)
        return view_func(*args, **kwargs)
    return wrapper
```

Это важно, потому что API-клиент ожидает JSON-ответ, а не HTML-редирект.

## Ошибка при отсутствии авторизации

Если неавторизованный пользователь попытается создать, изменить или удалить рецепт, API вернёт:
 `401 Unauthorized`

Ответ:

```json
{
  "success": false,
  "error": "Необходима авторизация."
}
```

## Проверка прав доступа

Авторизация отвечает на вопрос: вошёл ли пользователь в систему.

Но этого недостаточно. Нужно также проверить, имеет ли пользователь право выполнять действие над конкретным объектом.

Например, пользователь может редактировать только собственный рецепт.

Проверка выполняется на сервере:

```python
if recipe.author_id != current_user.id:
    return api_error("Нет прав на изменение этого рецепта.", 403)

```

Если пользователь пытается изменить чужой рецепт, API возвращает: `403 Forbidden`

Ответ:

```json
{
  "success": false,
  "error": "Нет прав на изменение этого рецепта."
}
```

## Почему не используется JWT

В текущей версии проекта API встроен в Flask-приложение, которое уже использует Flask-Login для пользовательского интерфейса. Поэтому для учебного проекта логично использовать одну общую систему авторизации.

JWT или token-based authentication можно добавить в будущем, если API будет использоваться отдельным frontend-приложением, мобильным приложением или внешними клиентами.

## Как тестировать защищённые endpoint-ы

Так как API использует cookie-сессию, для тестирования защищённых endpoint-ов нужно быть авторизованным.

Есть несколько вариантов:

### Вариант 1. Через браузер
1. Открыть сайт. 
2. Войти в аккаунт. 
3. Открыть DevTools. 
4. Найти cookie session. 
5. Использовать эту cookie в Postman или другом API-клиенте.

### Вариант 2. Через Postman

В Postman можно добавить cookie:

`session=<значение session cookie>`

После этого запросы к защищённым endpoint-ам будут восприниматься сервером как авторизованные.

### Вариант 3. Через curl
Можно использовать файл cookies:

```text
curl -c cookies.txt -X POST http://127.0.0.1:5050/auth \
  -d "username=..." \
  -d "password=..."
```

## Ограничения текущего подхода

Session-based authentication хорошо подходит для текущего Flask-приложения, но имеет ограничения:

- API сложнее использовать внешним клиентам без браузерной сессии;
- для мобильного приложения удобнее token-based authentication;
- для публичного API лучше использовать отдельную схему токенов.

## Возможное развитие

В будущем можно добавить:

- JWT-аутентификацию;
- endpoint /api/auth/login;
- endpoint /api/auth/logout;
- refresh-токены;
- отдельные API-ключи.

## Итог

В текущей версии API Be cooking использует session cookie Flask-Login. Это позволяет переиспользовать уже существующую систему входа и ролей, а также защищать операции создания, редактирования и удаления рецептов.

# Нормализация базы данных

База данных проекта Be cooking спроектирована вокруг основных сущностей предметной области: пользователей, рецептов, шагов приготовления, комментариев, избранного, корзины, статей и поваров.

В большинстве случаев структура базы данных следует принципам нормализации: данные разделены на логические таблицы, связи между сущностями выражены через внешние ключи, а повторяющиеся связи вынесены в отдельные таблицы.

При этом в проекте осознанно используются отдельные элементы денормализации: ингредиенты рецепта хранятся как `array(JSON)`, а теги повара — как `array(string)`. Это сделано для упрощения структуры и уменьшения количества лишних join-запросов.

## Осознанная денормализация

Несмотря на общую нормализованную структуру, в проекте есть два осознанных отступления от строгой нормализации:

- ингредиенты рецепта хранятся в поле `ingredients` как JSON-массив;
- теги повара хранятся в поле `tags` как JSON-массив строк.

Эти решения приняты осознанно, с учётом масштаба учебного проекта и особенностей использования данных.

### Денормализация ингредиентов рецепта

В модели `Recipe` ингредиенты хранятся так:

```python
ingredients = db.Column(db.JSON)
```

**Пример структуры:**

```json
[
  {
    "name": "Мука",
    "amount": "200",
    "unit": "г"
  },
  {
    "name": "Яйцо",
    "amount": "2",
    "unit": "шт"
  }
]
```

**Почему ингредиенты не вынесены в отдельные таблицы**

Строго нормализованный вариант мог бы выглядеть так:

`recipe 1 ─── * recipe_ingredient * ─── 1 ingredient`

Но для текущей версии проекта это усложнило бы структуру:

- пришлось бы создавать отдельный справочник ингредиентов;
- нужно было бы обрабатывать синонимы и разные написания ингредиентов;
- потребовались бы дополнительные join-запросы;
- форма создания рецепта стала бы сложнее;
- для простого отображения рецепта пришлось бы собирать данные из нескольких таблиц.

В текущей логике ингредиенты почти всегда используются как часть конкретного рецепта. Они не редактируются отдельно от рецепта и не имеют собственной страницы или независимой бизнес-логики.

Поэтому хранение ингредиентов как JSON-массива упрощает работу с формой рецепта и ускоряет чтение детальной страницы рецепта.

**Преимущества такого подхода:**
- простая форма создания и редактирования рецепта;
- меньше таблиц в базе данных;
- меньше join-запросов при отображении рецепта;
- ингредиенты сразу доступны как часть объекта рецепта;
- удобно пересчитывать ингредиенты для продуктовой корзины.

**Ограничения такого подхода:**

- сложнее искать рецепты по конкретному ингредиенту;
- сложнее агрегировать статистику по ингредиентам;
- нет единого справочника ингредиентов;
- возможны разные написания одного ингредиента: помидор, помидоры, томат;
- сложнее обеспечить строгую целостность данных.

Для текущего проекта это допустимый компромисс, потому что ингредиенты используются преимущественно внутри рецепта, а не как самостоятельная сущность.

В будущем, если появятся функции поиска по ингредиентам, подбора рецептов по продуктам или аналитики, ингредиенты можно будет вынести в отдельные таблицы.

### Денормализация тегов поваров

В модели `Chef` теги хранятся так:

```python
tags = db.Column(db.JSON)
```

Пример:

`["итальянская кухня", "французская кухня", "десерты"]`

**Почему теги не вынесены в отдельную таблицу**

Строго нормализованный вариант мог бы включать таблицы:
- Tag
- ChefTag

Но в текущей версии проекта теги используются только для отображения категорий на карточке повара. У них нет отдельной страницы, сложной фильтрации или независимого управления.

**Преимущества:**
- простая реализация;
- быстрое чтение данных;
- удобное отображение тегов в шаблоне;
- меньше служебных таблиц.

**Ограничения:**
- нет централизованного справочника тегов;
- возможны дубли и разные написания;
- сложнее сделать фильтрацию по тегам;
- сложнее переименовать тег сразу у всех поваров.

Если в будущем понадобится полноценная система категорий или фильтров, теги можно будет нормализовать и вынести в отдельные таблицы.

## Почему денормализация здесь допустима

Денормализация допустима, когда она применяется осознанно и решает конкретную задачу.

В этом проекте она используется для данных, которые:

- тесно связаны с родительской сущностью;
- не имеют самостоятельного жизненного цикла;
- редко используются отдельно;
- чаще всего отображаются вместе с родительским объектом.

Ингредиенты существуют внутри рецепта.
Теги существуют внутри карточки повара.

Поэтому для текущего масштаба проекта JSON-поля являются практичным компромиссом между строгостью модели и простотой реализации.

## Итог

База данных проекта в основном следует принципам нормализации:

- сущности разделены на отдельные таблицы;
- связи реализованы через внешние ключи;
- многие-ко-многим связи вынесены в промежуточные таблицы;
- для пользовательских данных реализованы ограничения уникальности.

При этом проект осознанно использует ограниченную денормализацию:

- Recipe.ingredients хранит массив JSON-объектов;
- Chef.tags хранит массив строк.

Это сделано для упрощения структуры, уменьшения количества join-запросов и более удобной работы с формами. Для текущего масштаба проекта такое решение оправдано, а при дальнейшем развитии системы может быть заменено на более строгую нормализованную модель.

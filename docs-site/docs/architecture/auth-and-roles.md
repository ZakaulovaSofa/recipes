# Аутентификация и роли пользователей

В проекте Be cooking реализована система аутентификации пользователей и разграничения доступа на основе ролей. Это необходимо для защиты пользовательских данных, ограничения доступа к административным функциям и предотвращения изменения чужого контента.

## Общая логика аутентификации

Пользователь может создать аккаунт или войти в уже существующий через единую форму авторизации. Если пользователь с указанным логином уже существует, система выполняет вход. Если пользователя с таким логином нет, создаётся новый аккаунт.

Аутентификация реализована с использованием Flask-Login. После успешного входа пользователь сохраняется в сессии, а в шаблонах и маршрутах становится доступен объект `current_user`.

Основные возможности:

- регистрация нового пользователя;
- вход существующего пользователя;
- выход из аккаунта;
- проверка текущего пользователя через `current_user`;
- защита маршрутов через декоратор `@login_required`.

## Хранение паролей

Пароли пользователей не хранятся в базе данных в открытом виде. При регистрации пароль хешируется с помощью Werkzeug:

```python
generate_password_hash(password)
```

При входе введённый пароль сравнивается с сохранённым хешем:

```python
check_password_hash(self.password_hash, password)
```

Это защищает пользователей от утечки исходных паролей в случае компрометации базы данных.

## Модель пользователя

Пользователь представлен моделью `User`.

**Основные поля:**

- `username` – логин пользователя
- `password_hash` – хеш пароля
- `avatar_url` – ссылка на аватар
- `created_at` – дата создания аккаунта
- `role` – роль пользователя

Роль пользователя хранится через перечисление `UserRoleEnum`.

```python
class UserRoleEnum(enum.Enum):
    USER = "user"
    ADMIN = "admin"
```

## Роли в системе

В проекте предусмотрены две роли:

- `USER` – обычный пользователь сайта
- `ADMIN` – администратор с расширенными правами

### USER

Обычный пользователь **может**:
- создавать рецепты;
- редактировать и удалять свои рецепты;
- добавлять рецепты в избранное;
- вести заметки к рецептам;
- добавлять рецепты в продуктовую корзину;
- оставлять комментарии;
- редактировать и удалять свои комментарии;
- изменять собственный профиль и аватар.

Пользователь **не может**:
- редактировать чужие рецепты;
- удалять чужие рецепты;
- управлять административными материалами;
- заходить в административную панель;
- выкладывать статьи;
- выкладывать информацию о поварах.


### ADMIN

Администратор **может**:

- пользоваться Flask-Admin панелью;
- управлять пользователями;
- управлять рецептами;
- управлять статьями;
- управлять карточками поваров;
- модерировать содержимое системы;
- при необходимости назначать других пользователей администраторами.

Администраторская роль нужна для управления справочными и публичными материалами сайта.

## Защита маршрутов

Для страниц, доступных только авторизованным пользователям, используется декоратор `@login_required`.

Например, доступ к профилю, созданию рецепта, управлению своими рецептами, корзине и избранному должен быть доступен только после входа.

Пример:

```python
@app.route('/profile')
@login_required
def profile():
    ...
```

Если неавторизованный пользователь пытается открыть защищённый маршрут, Flask-Login перенаправляет его на страницу авторизации.

## Защита данных пользователя

В проекте реализована дополнительная проверка владельца данных. Это важно, потому что одного @login_required недостаточно: он проверяет только факт входа, но не проверяет, имеет ли пользователь право работать с конкретным объектом.

Например, при открытии страницы управления рецептом проверяется, что текущий пользователь является автором рецепта:

```python
if recipe.author_id != current_user.id:
    abort(403)
```

Такая проверка защищает от прямого перехода по URL чужого рецепта, например:
`/recipes/15/manage`

Если рецепт принадлежит другому пользователю, сервер вернёт ошибку 403 Forbidden.

## Доступ к административной панели

Flask-Admin панель должна быть доступна только пользователям с ролью ADMIN.

Для этого используется защищённый класс представления, который проверяет:

- пользователь авторизован;
- роль пользователя — ADMIN.

Пример логики:

```python
current_user.is_authenticated and current_user.role == UserRoleEnum.ADMIN
```

Если пользователь не является администратором, доступ к административному интерфейсу запрещается.

## Итог

В Be cooking реализована базовая, но полноценная система аутентификации и управления доступом:

- пароли хранятся в виде хешей;
- вход и выход реализованы через Flask-Login;
- защищённые страницы закрыты декоратором `@login_required`;
- права на изменение объектов проверяются на сервере;
- пользователи разделены на роли USER и ADMIN;
- административные функции доступны только администраторам.
